# 边界行为决策

本文档冻结真实使用中容易出问题的边界行为。原则是：遇到高风险、不确定、外部状态变化或多目标歧义时，系统宁可追问或展示选择，也不得猜测或静默执行。

## 1. 日程冲突与创建

### 1.0 日程必须先形成本地草稿

聊天 agent 不得直接写 Google Calendar。用户提出创建或修改日程时，
系统必须先创建或更新本地 `calendar_event_draft` Artifact，并在 assistant
回复文本中列出关键日程内容。只有用户明确确认后，才允许通过执行工具创建真实日程。

补充：日程页是用户手动操作界面，不属于 Assistant 代办。用户在日程页点击
新建、保存修改或删除按钮时，视为明确的人为操作，可以直接写入 Google
Calendar。这个直接写入能力不得被 Assistant 工具绕过使用；Assistant 仍必须
走本地草稿、Proposal、用户确认和 Execution Service。

### 1.1 创建前必须检查用户自己的目标日历

创建或更新 `calendar_event` 前，系统必须对用户自己的目标日历执行 `Freebusy` 查询。

如果无法查询用户自己的目标日历忙闲状态：

```text
不得进入 create_calendar_event / update_calendar_event 的可执行 Proposal。
必须提示用户重新授权、选择其他日历，或稍后重试。
```

### 1.2 发现冲突时的默认行为

如果目标时间与用户自己的日历冲突：

```text
默认不得直接创建普通 Proposal。
系统必须展示冲突摘要，并提供下一步选择。
```

冲突摘要至少包含：

- 冲突事件标题，必要时脱敏；
- 冲突开始和结束时间；
- 目标日历；
- 当前请求的会议时间；
- 是否包含会议缓冲时间冲突。

### 1.3 替代时间建议

发现冲突后，系统应在以下范围内给出 2 到 3 个替代时间：

```text
优先当天剩余工作时间；
其次未来 3 个工作日；
默认避开午休；
默认应用 meeting_buffer_minutes；
默认不跨越用户工作时间边界。
```

如果找不到替代时间，系统应说明原因，并询问用户是否扩大搜索范围。

### 1.4 用户强制创建冲突日程

用户可以明确要求仍然创建冲突日程。

要求：

- 用户必须使用明确表达，例如“仍然创建”“忽略这个冲突创建”“我知道冲突，继续创建”；
- Proposal payload 必须包含 `conflict_override=true`；
- assistant 回复文本和 Proposal 信息必须展示冲突信息；
- 执行前仍需再次检查用户自己的日历。

### 1.5 执行前二次检查

用户确认后，`Execution Service` 在真正调用 Calendar API 前必须再次执行 `Freebusy`。

如果二次检查发现新的冲突：

- 没有 `conflict_override=true` 时，停止执行，Proposal 回到需要用户处理的状态；
- 已有 `conflict_override=true` 但冲突对象变化时，必须重新展示并再次确认；
- 如果冲突无变化，可以继续执行。

### 1.6 外部参会人 Freebusy 不可读

无法读取外部参会人的忙闲状态时，不阻止创建 Proposal，但必须在 assistant 回复文本中展示：

```text
你的日历在该时段没有冲突。
我无法读取部分参会人的忙闲状态，他们是否有空仍需发送邀请后确认。
```

### 1.7 重复会议

MVP 中重复会议必须满足：

- 用户明确提出重复规则；
- 系统能解析为合法 recurrence rule；
- assistant 回复文本或 Proposal 信息展示重复规则的人类可读说明。

无法可靠解析重复规则时，不得进入 Proposal。

## 2. 时间、时区与自然语言时间

### 2.1 相对时间必须保存解析基准

解析“明天”“下周五”“下午三点”等相对时间时，必须保存：

```text
original_text
base_time
timezone
resolved_start
resolved_end 或 duration_minutes
source_type
```

### 2.2 模糊时间不得直接执行

以下表达不能直接进入可执行 Proposal：

```text
下午
下班前
晚点
这周找个时间
有空的时候
```

系统可以先生成建议时间，但必须让用户确认。

### 2.3 夏令时和不存在时间

如果目标时区存在夏令时切换，导致时间不存在或重复：

- 不得猜测；
- 必须展示候选解释；
- 由用户选择具体时间。

## 3. 邮件回复与发送

### 3.1 reply 与 reply_all

默认回复策略：

```text
用户说“回复他/她”：reply sender
用户说“回复所有人/大家都回”：reply_all
用户选中邮件且原邮件有多个收件人：默认 reply sender，并在 assistant 回复文本中提示可改为 reply_all
```

如果用户文本和选中上下文冲突，必须追问。

### 3.2 引用原文

MVP 默认不把完整原邮件正文复制进本地草稿。

发送时是否保留 Gmail 原生 quoted content 由 Gmail reply 流程决定。assistant 回复文本只展示助理新写的正文和必要上下文摘要。

### 3.3 HTML 与纯文本

MVP 本地 Artifact 保存：

```text
body_text
body_html 可选
```

如果没有可靠 HTML 模板，发送时使用纯文本正文。不得把未经清洗的外部 HTML 直接拼入邮件。

### 3.4 邮件发送失败后的恢复

如果 Gmail Draft 已创建但发送失败：

- 记录 `external_resource_id`；
- Proposal 状态进入 `failed` 或 `execution_unknown`；
- 前端提示用户可重试发送该 Draft 或取消；
- 不得重新创建第二个 Gmail Draft，除非确认原 Draft 不存在。

### 3.5 “发给他们”的范围

“他们”“这些人”“所有人”等集合指代必须来自：

- 用户本轮明确列出的人；
- 当前选中邮件/thread 的明确参与者；
- 当前 Work Item 已解析的收件人列表。

候选集合不唯一时必须追问。

## 4. 文件上传与解析失败

### 4.1 OCR

MVP 不默认支持扫描 PDF OCR。

如果 PDF 没有可提取文本：

```text
提示用户该文件可能是扫描件，当前无法解析文字。
不得假装已经读取内容。
```

### 4.2 加密或损坏文件

加密 PDF、损坏 DOCX、无法识别格式：

- `file_extractions.status=failed`；
- 保存错误类型；
- 向用户说明无法解析；
- 不得把文件作为内容依据进入 Proposal。

### 4.3 文件删除

文件删除后：

- 新任务不得继续引用该文件解析文本；
- 已经生成的 Artifact 可以保留当时提取出的必要片段和 evidence 摘要；
- Proposal 卡片必须继续展示其来源来自已删除文件，并提醒用户来源文件不可再打开。

### 4.4 文件去重

相同 `sha256` 的文件重复上传时：

- 可复用已有解析结果；
- 仍需记录本次上传与当前 thread 的关联；
- 不得把其他用户的同 hash 文件暴露给当前用户。

### 4.5 文件隐私

文件内容不得自动写入长期记忆。

用户明确说“记住这个偏好”时，只能保存被用户明确指定的偏好，不保存整段文件内容。

## 5. 选中上下文冲突

### 5.1 选中对象与文本冲突

如果用户选中 A 邮件，但文本说“回复李明那封”，而 A 邮件不是李明相关：

```text
必须追问：你是要处理当前选中的邮件，还是李明那封？
```

### 5.2 多选邮件

用户多选邮件后提出请求：

- “总结这些邮件”：允许合并总结；
- “分别回复”：创建多个 Work Item；
- “回复他们”：必须确认每封邮件的回复目标；
- 意图不明确时追问。

### 5.3 文件片段优先级

同时选中文件片段和整份文件时：

```text
默认优先使用选中文件片段。
只有用户要求“看完整文件”时，才扩展到整份文件。
```

### 5.4 刷新后的选中状态

页面刷新后，前端可以恢复视觉选中状态，但每次聊天请求仍必须重新发送 `selected_context_refs`。

后端不得使用上一次请求的 selected context 作为本轮权威目标。

## 6. 授权、执行与恢复

### 6.1 Proposal 过期

MVP 默认 Proposal 有效期为 24 小时。

过期后：

- 不能执行；
- 需要重新生成 Proposal；
- 重新计算 version 和 fingerprint。

### 6.2 Token 过期或撤销

如果执行时 Google Token 过期：

- 能刷新则刷新后继续；
- 不能刷新则停止执行；
- Proposal 保留为待处理；
- 前端提示用户重新连接 Google 账号。

如果用户撤销 Google 授权：

- Open Work Item 保留；
- 不能进入外部写执行；
- 重新授权后必须再次校验 Proposal 是否仍然最新。

### 6.3 多浏览器标签重复确认

多个浏览器标签同时确认同一 Proposal 时：

- 通过数据库条件更新获得执行权；
- 只有一个请求可以从 `approved` 进入 `executing`；
- 其他请求返回已有状态或执行结果。

### 6.4 execution_unknown 展示

当状态为 `execution_unknown` 时，普通用户界面应展示：

```text
我无法确认刚才的外部操作是否已经完成。为避免重复发送或重复创建，我不会自动重试。
你可以打开 Gmail/Calendar 检查结果，或选择让我重新同步状态。
```

不得展示内部错误堆栈。

## 7. OAuth 与权限

### 7.1 Scope 原则

MVP 可先使用能覆盖功能的 scope，但文档和配置必须保留最小权限路线。

如果只需要撰写和发送邮件，优先评估：

```text
gmail.compose
```

只有确实需要修改邮箱状态时，才使用更大的 scope。

### 7.2 前端连接入口

前端必须提供 Google 连接状态和操作入口，但不得要求用户在前端填写 OAuth Client ID 或 Client Secret。

用户可见状态：

```text
未连接 Google
正在跳转授权
授权成功
授权失败
已连接账号
需要重新连接
```

用户可见操作：

```text
连接 Google
重新连接
断开连接
```

前端调用后端 OAuth API 完成授权流程。`GOOGLE_CLIENT_ID`、`GOOGLE_CLIENT_SECRET`、`GOOGLE_REDIRECT_URI` 属于后端 `.env` 配置，其中 `GOOGLE_CLIENT_SECRET` 不得返回给前端。

### 7.3 切换 Google 账号

用户切换 Google 账号后：

- 旧账号相关 Proposal 不得继续执行；
- Work Item 可保留为本地草稿；
- 必须重新确认发件账号、目标日历和权限。

## 8. 长期记忆

### 8.1 写入长期记忆需要明确意图

以下表达可以写入长期记忆候选：

```text
以后都...
记住...
默认...
以后会议都...
```

以下表达只作用于当前 Work Item：

```text
这次...
这封...
这个会议...
今天先...
```

### 8.2 记忆与 user_settings 冲突

冲突时优先级：

```text
用户本轮明确输入
→ user_settings
→ 已激活长期记忆
→ 系统默认值
```

`user_settings` 是确定性设置的事实来源，长期记忆不能静默覆盖它。

### 8.3 查看和删除记忆

MVP 至少需要提供后端能力，让用户查看、删除长期记忆。

没有查看/删除能力前，不应自动保存高敏感偏好。

## 9. 交互追问

### 9.1 一次追问的字段数量

一次追问默认最多包含 3 个问题。

超过 3 个时：

- 优先问阻塞 Proposal 的必填字段；
- 低风险偏好可以放到卡片中稍后确认。

### 9.2 短确认语解析

“确认”“可以”“发吧”“就这样”必须结合候选动作类型解析。

如果当前存在多个不同动作类型的 Proposal：

- “发吧”只筛选 `send_email`；
- “创建吧”只筛选 `create_calendar_event`；
- “确认”在多个动作候选存在时必须追问；
- “全部确认”涉及多个外部写操作时必须展示范围并二次确认。

聊天 agent 的即时确认工具命名规则：

- “确认发送”调用 `send_email_draft`；
- “创建日程”“确认创建日程”调用 `execute_calendar_event_draft`；
- 选中已有日程后说“改到/修改/加参会人”时，先调用 `create_calendar_update_draft`；
- 确认更新已有日程时调用 `execute_calendar_event_update_draft`；
- 如果存在 active draft，补充信息必须更新原草稿，不得重复创建；
- 如果有日程冲突，用户必须明确“仍然创建”才允许带 `conflict_override=true` 执行。

### 9.3 卡片信息密度

确认卡片必须展示会影响外部行为的字段，但不展示完整内部 evidence。

默认展示：

- 收件人/参会人；
- 主题/标题；
- 正文或描述摘要；
- 时间、时区、日历；
- 冲突状态；
- 文件或邮件来源摘要；
- version 和风险提示。

详细 evidence 放入展开区域。

## 10. 隐私、日志与保留

### 10.1 邮件正文入库

MVP 可以保存生成的本地草稿正文。

读取到的外部邮件正文默认只保存必要摘要、message id、thread id 和 evidence 引用。除非用户明确要求，不保存完整历史邮件正文。

### 10.2 上传文件保留

MVP 默认上传文件保留到用户删除。

后续可增加自动清理策略，但清理前必须确保相关 Artifact 的 evidence 摘要仍可解释。

### 10.3 Markdown 导出

Markdown 导出不得包含：

- Token；
- OAuth secret；
- 完整敏感邮件正文；
- 完整上传文件文本；
- Proposal 授权唯一事实；
- 执行状态唯一事实。
